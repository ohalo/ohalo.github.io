上周五，运维工程师老周跑来问我："能不能让AI帮我们查查故障库？每次新建工单都要翻半天。"

我心想这不难——公司十年的故障记录都存在Confluence里，直接扔给大模型不就行了？

结果一测，第一天：✅ 回答得不错。
第二天：❓ 开始瞎编了。
第三天：🔥 直接报错，说知识库"没有访问权限"（其实是上下文窗口不够）。

这不是大模型的问题。是我根本没搞清楚：RAG不只是"把文档丢给AI"这么简单。

RAG的真相：不是"投喂"，是"外挂"

大多数程序员第一次接触RAG，会想当然地理解成：

上传文档 → 大模型学会 → 回答问题

大错特错。

RAG的本质是：在模型"思考"之前，先从你的文档里查一段上下文，夹在问题里一起发过去。

整个过程分三步：

第一步：存（索引阶段）

你的文档不会"进入"模型——它们被切成小块（chunk），每块转成一串数字（向量），存进向量数据库。

文档 → 切分 → 向量化 → 入库

这一步只需要做一次，后续只增不减。

第二步：查（检索阶段）

用户提问时，问题本身也被向量化，然后在向量数据库里"翻相似的内容"：

问题 → 向量化 → 相似度搜索 → 召回Top-K文档块

这一步是实时的，每次问答都会跑。

第三步：拼（生成阶段）

把召回的文档块 + 用户问题打包成prompt，扔给大模型：

用户问题 + 检索到的上下文 → 大模型 → 回答

模型看到的不再是"它训练时学过的东西"，而是"你文档里的内容"。

我踩过的三个坑

坑一：文档切分毁了检索质量

一开始我把PDF切成500字一块，发现检索结果总是"差一截"——用户问"XX模块的部署步骤"，召回的内容要么在"环境准备"阶段，要么已经到了"验收测试"，偏偏漏了最关键的部署命令。

根因：跨段落的语义被打断了。

后来改成滑动窗口切分（重叠200字）+ 保留段落结构，召回质量立刻提升。

具体做法：

  • 按标题层级切（## 作为切分边界）
  • 每块包含完整上下文
  • 块之间留15-20%的重叠

坑二：Embedding模型选错了

一开始图省事，用的是通用Embedding模型（训练数据是维基百科和新闻），结果：

  • 技术术语匹配率只有40%
  • "Pod"被理解成"容器"（字面意思）
  • "OOM"根本匹配不到任何内容

后来换成专门微调过的技术文档Embedding，术语匹配率提到85%+。

选Embedding模型的参考维度：

  • 文档类型（通用 vs 垂直领域）
  • 语言（中英文混合文档要选支持多语言的）
  • 块大小（不同模型支持的向量维度上限不同）

坑三：检索结果干扰了生成

有一次，用户问的是"故障等级怎么定"，召回的第一条是"故障处理SOP"，第二条是"故障等级定义"，第三条是"故障报告模板"——三条内容叠加，模型直接输出了一篇八股文，把三个文档的内容拼在一起了，完全答非所问。

解决思路：重排序（Rerank）+ 上下文压缩。

不是把所有召回内容都塞给模型，而是：

  1. 召回阶段多取几条（比如Top 20）
  2. 用重排序模型筛出最相关的前3-5条
  3. 必要时用压缩模型把每条内容精简到关键句子

三个实战方案，按难度排序

方案一：最快上手（AnythingLLM / Dify）

适合：快速验证想法，不写代码。

AnythingLLM支持拖拽上传PDF/Word/Markdown，自动切分+Embedding，点点鼠标就能跑起来。

Dify内置了完整的RAG pipeline，支持多种Embedding模型和重排序，还能接不同的向量数据库（Milvus / Qdrant / pgvector）。

缺点：定制化程度有限，大规模数据时性能会下降。

方案二：经典LangChain + 自托管向量库

适合：有开发能力，想自己掌控全链路。

核心组件：

文档 → Loader → Splitter → Embedding API → 向量库 → Retriever → Prompt拼接 → LLM API

这是我目前在生产环境跑的那套，主要参数：

  • 切分：按##标题层级，每块约800字，重叠150字
  • 向量库：Qdrant（支持按metadata过滤，查询性能比Milvus好）
  • Embedding：智谱的embedding-3，支持中文技术文档
  • LLM：GPT-4o-mini（日均调用3000次，成本可接受）

方案三：离线方案（不怕数据出境的可以用）

适合：公司对数据安全有要求，必须私有化部署。

关键组件换成开源版：

  • Embedding：Nomic Embed Text（4B参数，支持本地推理）
  • 向量库：Milvus / Qdrant（都可以docker一键部署）
  • LLM：Qwen2.5-7B / GLM4-9B（在Mac Studio M4 Ultra上能跑）

实测：M4 Ultra 192GB内存，跑7B模型+本地Embedding，响应延迟3-5秒，完全能接受。

最后说一句

RAG不是什么魔法，就是"先查再答"。核心难点不在于搭框架，而在于：

  • 文档怎么切（切得对才能查得准）
  • 向量怎么选（选得对才能匹配到）
  • 上下文怎么控（塞太多反而干扰模型）

搞明白这三点，你的知识库系统才算真正落地。
